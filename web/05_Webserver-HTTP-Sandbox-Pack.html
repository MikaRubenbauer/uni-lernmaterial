Nennen 4a — Zwei Aufgaben eines Webservers

1. HTTP-Requests entgegennehmen und beantworten
Auf TCP-Verbindung lauschen, HTTP-Anfragen parsen, passende HTTP-Response zurücksenden.
2. Ressourcen ausliefern
Statische Dateien (HTML, Bilder, CSS) direkt liefern ODER an Programmcode delegieren (dynamische Generierung).

Weitere mögliche Antworten: Zugriffe protokollieren (Logfiles), Authentifizierung/Zugriffsschutz, Virtuelle Hosts verwalten.

Nennen + Beschreiben 4b — Prozessmodelle

Frage variiert: „im selben Prozess" vs. „getrennte Prozesse" vs. „in welchem Modell wird CGI verwendet".

Selber Prozess (In-Process / Modul)

Webserver-Aufgabe und Programmcode laufen im gleichen Prozess (z.B. Server-Modul, eingebetteter Interpreter, oder NodeJS selbst).

+ schnell kein Prozess-Start pro Request.
– Risiko Crash im Code kann den Server mitreißen; weniger Isolation.

Getrennte Prozesse (CGI)

Pro Request startet der Server einen eigenen, neuen Prozess (Common Gateway Interface), der HTML erzeugt und zurückgibt.

+ Isolation Crash betrifft nur den einen Prozess; jede Sprache nutzbar.
– langsam Prozesserzeugung pro Anfrage kostet Performance.
Direkt-Antworten: „getrennte Prozesse" → CGI (neuer Prozess pro Request). „selber Prozess" → Modul/In-Process (z.B. Server-Modul, NodeJS-Eventloop).

Erstellen N1 — Minimaler NodeJS-Server (http-Modul) + npm

const http = require("http");                  // eingebautes Modul
const server = http.createServer((req, res) => {
  res.writeHead(200, {"Content-Type": "text/html"});
  res.end("Hello World");
});
server.listen(3000);                          // lauscht auf Port 3000

Verifiziert: liefert 200 | Hello World. npm = Node Package Manager: npm init (package.json), npm install express (Abhängigkeit installieren), Skripte via npm run.

Dasselbe mit Express (kürzer) 
const express = require("express");
const app = express();
app.get("/", (req, res) => res.send("Hello World"));
app.listen(3000);

Nennen / Beschreiben 3a — JavaScript-Sandbox

Was die Sandbox verhindert (Beispiele)

  • Kein direkter Dateisystem-Zugriff (lokale Dateien lesen/schreiben).
  • Kein Zugriff auf andere Origins (Same-Origin-Policy) — fremde Domains/Tabs.
  • Keine Betriebssystem-/Hardware-Befehle.

Schutzfunktion (Beschreibung)

JavaScript läuft isoliert im Browser-Kontext einer Seite. Es darf nur mit dem DOM dieser Seite und erlaubten Web-APIs interagieren, aber nicht auf das System des Nutzers oder Daten anderer Websites zugreifen → schützt vor schädlichem Code.

Konkretes Beispiel für die Klausur: „Eine Webseite kann per JS NICHT die Datei C:\\passwoerter.txt vom Rechner lesen" oder „kann nicht die Cookies/Daten einer fremden Domain auslesen (Same-Origin-Policy)".
