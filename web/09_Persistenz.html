Persistenz (N8)

Restthema — Folien „NodeJS Day4: Persistenz". Daten über den einzelnen Request hinaus speichern.

Konzept Warum Persistenz?

Variablen im Server-Code sind flüchtig — bei Neustart weg, und nicht zwischen Nutzern geteilt. Persistenz speichert Daten dauerhaft: in Dateien oder einer Datenbank.

ArtWannBeispiel
Datei (z.B. JSON)kleine Datenmengen, einfachfs.writeFile
Datenbankviele/strukturierte Daten, AbfragenSQLite, PostgreSQL, MongoDB

Erstellen Datei-Persistenz mit fs

Schreiben

const fs = require("fs");
const daten = { count: 5 };
fs.writeFileSync("data.json",
  JSON.stringify(daten));

Lesen

const roh = fs.readFileSync("data.json", "utf-8");
const daten = JSON.parse(roh);
console.log(daten.count); // 5
Sync vs. Async: writeFileSync blockiert (einfach, für kleine Skripte). In echten Servern besser asynchron (fs.promises.writeFile + await), damit der Event-Loop nicht blockiert.

Erstellen Datenbank (Skizze, SQLite)

// Konzept — pro DB-Lib unterschiedlich
db.run("INSERT INTO besucher (name) VALUES (?)", [name]);   // schreiben (Prepared Statement!)
db.all("SELECT * FROM besucher", (err, rows) => { … });   // lesen

Platzhalter ? = Prepared Statement → schützt vor SQL-Injection (siehe Sicherheit-Restthema).

Drill

[Beschreiben] Warum reicht eine normale Variable im Server nicht für Persistenz?

Sie ist flüchtig: bei Server-Neustart verloren und nicht dauerhaft gespeichert. Persistenz braucht Datei oder Datenbank.

[Erstellen] Objekt {n:1} in eine Datei schreiben (NodeJS). 
require("fs").writeFileSync("d.json", JSON.stringify({n:1}));
[MC] Wofür steht das ? in einer SQL-Abfrage?

Prepared-Statement-Platzhalter — Wert wird sicher gebunden, schützt vor SQL-Injection.
