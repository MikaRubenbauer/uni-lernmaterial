Erstellen 5b — Middleware (param==42 → Redirect /spezial)

Aufgabe: Bei Anfrage auf /anfrage prüfen, ob der Body den Parameter param mit Wert 42 hat. Wenn ja → Weiterleitung zu /spezial, sonst normal weiterlaufen lassen.

// req.body braucht einen Body-Parser, z.B. davor: app.use(express.json())
app.use("/anfrage", (req, res, next) => {
  if (req.body.param == "42") {      // param vorhanden & == 42
    res.redirect("/spezial");       // → 302, Browser geht zu /spezial
  } else {
    next();                          // sonst: nächste Middleware/Route
  }
});
✅ Verhalten (gegen Logik geprüft)
BodyReaktion
{param:"42"}302 Redirect zu /spezial
{param:"99"}next() — normale Anfrage
{} (kein param)next() — normale Anfrage
Falle: Ohne next() im else-Zweig „hängt" die Anfrage (keine Antwort). Immer entweder antworten ODER next() aufrufen.
Falle: req.body ist nur befüllt, wenn ein Body-Parser aktiv ist (express.json() / express.urlencoded()). Laut Aufgabe „Body existiert immer" → vorausgesetzt.

Erklären 5g — Wie funktionieren Cookies technisch?

Browser
Server
1. Request
2. Response + Set-Cookie: id=abc
3. Request + Cookie: id=abc
🍪 speichert id=abc
Cookie-Ablauf Schritt für Schritt.

Ablauf: Server sendet im Response den Header Set-Cookie: name=wert → Browser speichert ihn → bei JEDEM passenden Folge-Request hängt der Browser automatisch Cookie: name=wert an.

Wozu: HTTP ist zustandslos (jeder Request unabhängig). Cookies erweitern das Request-Response-Paar um Zustand → Wiedererkennung (Login, Warenkorb, Session-ID).

Klausur-Kernsatz: Cookies machen das zustandslose HTTP zustandsbehaftet — Server setzt via Set-Cookie, Browser schickt automatisch zurück.

Sichere Attribute: HttpOnly (kein JS-Zugriff), Secure (nur HTTPS), SameSite (CSRF-Schutz).

[Erklären] Warum braucht man Cookies, obwohl es Request-Parameter gibt?

HTTP ist zustandslos; Parameter gelten nur für einen Request. Cookies werden vom Browser automatisch bei jedem Folge-Request mitgeschickt → dauerhafte Wiedererkennung ohne manuelles Mitschicken.
